    - 顶层 `SKILL.md` 与 `references/schema.md` / `references/queries.md` 视为「构建产物」，由 default_* 模板与扩展 YAML 合成，而不是事实来源。

    - 在 default_skill 模板中预留可由 helper 管理的块，例如：  
      - `<!-- helper:core-types:start --> ... <!-- helper:core-types:end -->`（核心类型声明，可在块内追加扩展类型）；  
      - `<!-- helper:constraints:start --> ... <!-- helper:constraints:end -->`（约束示例，可在块内合并扩展约束）；  
      - `<!-- helper:skill-contract:start --> ... <!-- helper:skill-contract:end -->`（Skill Contract 中的 reads/writes 等，可根据扩展类型更新）；  
      - `<!-- helper:extensions-section:start --> ... <!-- helper:extensions-section:end -->`（专门用于描述扩展能力的 section，由 helper 全量生成）。  
    - 在 default_schema 模板中预留扩展挂载点，例如：  
      - `<!-- helper:schema-types-ext:start --> ... <!-- helper:schema-types-ext:end -->`；  
      - `<!-- helper:schema-relations-ext:start --> ... <!-- helper:schema-relations-ext:end -->`；  
      - `<!-- helper:schema-constraints-ext:start --> ... <!-- helper:schema-constraints-ext:end -->`。  
    - 在 default_queries 模板中预留扩展示例挂载点，例如：  
      - `<!-- helper:queries-extensions:start --> ... <!-- helper:queries-extensions:end -->`，用于放置 lineage 等扩展相关的查询示例。  
  - 制定无扩展 / 有扩展两种生成模式：  
    - 无扩展时：顶层 `SKILL.md` 与 `references/schema.md` / `references/queries.md` 直接从 default_* 模板同步（仅复制模板内容，不追加扩展片段）；  
    - 有扩展时：helper 从 default_* 模板拷贝整体结构，仅在上述标记区块内部追加或合成由 YAML 扩展推导出的片段，保证默认描述与扩展描述在文档中有清晰分界。  
  - 扩展开关与执行路径设计：  
    - 扩展能力的「开关」优先放在 `ontology.py` 中，通过显式子命令触发（例如 `extend-lineage`，名称可根据后续设计微调）；  
    - 仅当用户主动调用该扩展子命令时，才认为「需要启用某类扩展」，在此之前 helper 不应自动读取或应用任何扩展 YAML；  
    - 扩展子命令的职责应包括两部分：  
      - 使用 `schema-append` 或等价机制，将扩展 YAML 合并进运行时使用的 `memory/ontology/schema.yaml`；  
      - 调用 helper 以 default_* 模板 + 扩展 YAML 为输入，重新生成顶层 `SKILL.md` 与 `references/schema.md` / `references/queries.md`；  
    - helper 与 `ontology.py` 的关系：  
      - `ontology.py` 作为入口脚本，负责对外暴露扩展相关命令（是否扩展、扩展哪一类）；  
      - helper 作为文档与 meta 的构建器，只在被显式调用时，根据扩展配置更新各 md 文档中带标记的结构化区块。  
  - 扩展 YAML 结构约定（以 lineage 扩展为例）：  
    - 顶层结构与运行时 schema 保持兼容，便于通过 `schema-append` 直接合并：  
      - `types`：扩展的实体类型定义（例如 `Table` / `Column` / `Job` 等），字段与 `schema.yaml` 中类型定义一致：  
        - `required`: [field1, field2, ...]  
        - `properties`: 按字段名列出类型与可选性（string / enum(...) / ref(...) / ...）。  
      - `relations`：扩展的关系定义（例如 `table_has_column` / `lineage` 等），字段与现有 relation 定义一致：  
        - `from_types` / `to_types` / `cardinality` / `acyclic` / `properties` 等。  
      - `constraints`：可选的扩展级约束，结构与默认 `constraints` 列表一致（支持 `type` / `relation` / `rule` / `message` / `enforcement` 等字段）。  
    - 为 helper 提供的文档与 meta 辅助字段：  
      - `meta`：helper 可选的文档辅助信息，不直接写回 `schema.yaml`：  
        - `docs`：描述 helper 在生成文档时使用的标题或小节名称（例如扩展类型小节名、扩展查询小节名等），属于「可有可无」的文案增强，而不是能力声明的唯一来源。  
      - `queries`：扩展专属查询示例列表，仅由 helper 用于生成 `references/queries.md` 与 `SKILL.md` 的相关示例：  
        - 每条示例可以约定字段：  
          - `id`：示例标识（如 `lineage_upstream`）；  
          - `title`：示例标题（如「查指定表的上游来源」）；  
          - `cli`：对应的一条 CLI 示例命令（如 `python3 scripts/ontology.py related ...`），可包含占位符；  
          - `description`：说明这条查询在什么场景下使用。  
    - 扩展 YAML 的整体定位：  
      - `types` / `relations` / `constraints` 用于运行时约束与校验，通过 `schema-append` 合并进入 `memory/ontology/schema.yaml`；  
      - `meta.docs` / `queries` 只被 helper 使用，用于驱动 `SKILL.md` 与 `references/*.md` 的结构化更新，不写入运行时 schema。  
  - SKILL frontmatter 描述的自动更新策略：  
    - `SKILL.md` 中的 `name` 视为稳定标识，不随扩展改变（仍然是 `ontology`）；  
    - `SKILL.md` 中的 `description` 由 helper 统一维护，其内容由两部分自动合成：  
      - 基础描述：来自 `references/default/default_skill.md` 中的默认 frontmatter 描述，概括 ontology 的基础图谱能力；  
      - 扩展描述：helper 在扩展生效时，基于已合并入 `schema.yaml` 的扩展 `types` / `relations` / `constraints` 自动推导出一段「扩展能力」描述（例如检测到有 `Table` / `Column` / `Job` 以及 `lineage` 关系时，自动追加“支持表/字段级数据血缘追踪与作业依赖分析”之类的短句）；  
    - 生成规则只依赖 schema 本身，不需要在扩展 YAML 里额外维护一份 `meta.skill` 描述，避免为一次扩展维护多份重复信息；  
    - 为了仍然满足「只改受控区域」的约束，可以在 `SKILL.md` frontmatter 中使用注释标记 description 为 helper 管理区域，例如：  
      - `# helper:description:start` / `# helper:description:end`，helper 仅在这两个标记之间更新 `description` 字段。  
  - `extend-lineage` 命令的对外契约（参数与行为）：  
    - 命令形式（示意）：  
      - 基本用法：`python3 scripts/ontology.py extend-lineage`  
      - 可选参数（仅设计约定，不代表已实现）：  
        - `--config/-c <path>`：指定扩展 YAML 文件路径，默认例如 `references/lineage/lineage.yaml`；  
        - `--schema-only`：只更新 `memory/ontology/schema.yaml`，不触碰任何 md 文档；  
        - `--docs-only`：只根据扩展 YAML + default_* 重新生成 `SKILL.md` 与 `references/*.md`，不修改运行时 schema；  
        - `--dry-run`：打印计划与预期变更摘要，不写入任何文件。  
    - 行为约束：  
      - 不调用 `extend-lineage` 时：  
        - 运行时的 `schema.yaml` 只包含默认与显式 `schema-append` 写入的内容；  
        - helper 在生成 `SKILL.md` / `references/*.md` 时只使用 default_* 模板，不加载扩展 YAML。  
      - 调用 `extend-lineage` 且未指定特殊模式（默认模式）：  
        - 先加载 `--config` 指定的扩展 YAML，从中取出 `types` / `relations` / `constraints` 片段，通过 `append_schema` 或等价逻辑合并进当前 `memory/ontology/schema.yaml`；  
        - 再调用 helper，以 default_* 模板 + 同一份扩展 YAML 为输入，重新生成顶层 `SKILL.md` 与 `references/schema.md` / `references/queries.md` 中带 helper 标记的区块；  
        - 生成过程遵循「只改 helper 标记区域、不改自然语言区域」的约束。  
      - 使用 `--schema-only` / `--docs-only` / `--dry-run` 等参数时：  
        - 仅执行对应的子流程（例如只更新 schema 或只更新文档），或仅输出计划不落盘，为后续实现保留扩展空间。  
  - helper 与扩展能力的分阶段落地计划：  
    - 阶段一（只动模板，不改逻辑）：  
      - 在 `references/default/default_skill.md` 中增加 helper 可管理的标记块（`core-types` / `constraints` / `skill-contract` / `extensions-section` 等）；

      - 在 `references/default/default_queries.md` 中增加查询扩展示例挂载点标记（`queries-extensions`）；  
      - 本阶段仅整理模板结构，不修改 `SKILL.md` / `references/schema.md` / `references/queries.md` 的现有语义。

- 存储后端 File / SQLite 端到端自测
  - File 存储后端（`KG_BACKEND=file`）  
    - 通过 `python -c` 直接导入 `ontology_core`，在同一进程内依次调用：  
      - `create_entity` / `get_entity` / `update_entity` / `list_entities` / `query_entities`；  
      - `create_relation` / `get_related`（`outgoing` + `incoming`）/ `delete_relation`；

    - 结论：在 FileStorage 后端下，实体/关系的 CRUD、查询与校验整条链路均可正常工作。  
  - SQLite 存储后端（`KG_BACKEND=sqlite`，`KG_DB_PATH=memory/ontology/test_kg.db`）  
    - 在同一模式下调用与 File 测试完全相同的一组 `ontology_core` 函数，底层通过环境变量切换为 `SQLiteStorage`。  
    - 观测结果：  
      - Person / Project 实体创建、更新、查询行为与 File 后端一致；  
      - 关系创建、查询（出边/入边）、取消关系的行为与 File 后端一致，`unrelate` 后不再出现对应边；  
      - `delete_entity` 返回 `True`，`validate_graph` 同样返回空错误列表 `[]`。  
    - 结论：在 SQLiteStorage 后端下，`BaseStorage` 抽象下挂的 SQLite 实现与 File 实现表现一致，`ontology_core` 公开方法在两种后端上均能完整跑通。

- 领域层与存储驱动抽象初步设计（更新）
  - 领域模型（Domain Layer）最终约定：  
    - `Entity`：`{ id, type, properties, created, updated }`  
      - `id`：全局唯一标识（如 task_xxx、table_xxx）；  
      - `type`：实体类型（如 `Task` / `Project` / `Table` 等）；  
      - `properties`：承载所有 schema 驱动的业务字段（status、priority、database、table 等），由 YAML schema + validate 约束；  
      - `created` / `updated`：领域层语义是“创建/更新时间”，内部可以用 `datetime` 表达，落盘时由适配层转为字符串。  
    - `Relation`：`{ from_id, type, to_id, properties }`  
      - `from_id` / `to_id`：表示一条有向边的起点/终点实体 ID；  
      - `type`：关系类型（如 `has_task` / `depends_on` / `lineage`），与 YAML 中 `relations` 段的 key 对齐；  
      - `properties`：承载关系上的附加信息（例如 lineage 边上的 edge_kind、confidence、source_job 等），同样由 schema 约束。  
  - 领域层职责边界：  
    - 只负责「图中的点和边」当前状态的结构定义，不直接包含 relations 列表（不在 `Entity` 中挂 `relations` 字段）；  
    - 不关心底层存储是文件 / SQLite / Supabase，也不关心 op log 的具体格式。  
  - 操作适配层（Adapter Layer）设计：  
    - 引入纯函数负责 `Entity` / `Relation` 与 operation 记录之间的互相转换，例如：  
      - `entity_to_create_op(entity, ts)` / `entity_to_update_op(entity_id, props, ts)`；  
      - `relation_to_relate_op(relation, ts)` / `relation_to_unrelate_op(from_id, type, to_id, ts)`；  
    - 这些函数集中定义 op 的结构规范（`op: "create" | "update" | "delete" | "relate" | "unrelate"` 以及字段映射），避免把存储细节散落在业务代码或抽象类中。  
  - 存储抽象层（BaseStorage & drivers）角色重新厘清：  
    - `BaseStorage`：保持为纯抽象，仅负责 `append_operation(op)` / `get_operations()` 两个方法；  
    - `FileStorage` / `SQLiteStorage` / `SupabaseStorage` / `SupabaseStorageRest`：  
      - 统一继承 `BaseStorage`，只实现“如何把 op 落到某种介质/从介质读出”的逻辑；  
      - 不直接依赖 `Entity` / `Relation`，也不关心 YAML schema 或血缘细节。  
  - 整体分层关系简要回顾：  
    - 领域层：`Entity` / `Relation` 描述图谱中的点和边；  
    - 适配层：负责领域对象与 op 日志之间的映射，实现「领域层 ↔ operation log」的解耦；  
    - 存储层：`BaseStorage` + 各种 drivers，只面向标准化的 operation 记录，不感知上层业务。  

- 代码现状审查与 ontology.py 拆分方向初步结论
  - 已对当前 `storage_backend.py`、`ontology.py` 以及两份文档进行了整体审查，结论如下：  
    - 架构方向基本正确：  
      - 统一的 `BaseStorage` 抽象，仅暴露 `append_operation` / `get_operations` 两个 op 级接口；  
      - `Entity` / `Relation` 作为领域模型已实际落地，并通过适配函数与 op 日志互相转换；  
      - `load_graph` 基于 op 回放构建内存态的实体与关系，再对外提供与旧实现兼容的 dict 视图；  
      - 多种存储后端（File / SQLite / Supabase / SupabaseRest）均通过统一的 op 协议工作。  
    - 存在的主要问题：  
      - `ontology.py` 职责过重：同时承载存储配置、领域适配、读写操作、校验逻辑、schema 工具与 CLI 解析，文件体量接近 700 行，远超「一个可读脚本」的简单度；  
      - 文档中的 Storage Interface 描述是基于实体级接口（`save_entity` / `get_entity` 等）的理想化设计，而当前实现采用的是 op 级 `BaseStorage`，两者存在一定落差，易造成理解偏差；  
      - 部分增强功能（如 SupabaseRest 与 SDK 双实现、`schema-append`、`unrelate` CLI 等）在整体架构尚未收敛前就落地，增加了认知与维护成本；  
      - 领域对象目前位于 `storage_backend.py` 中，从分层语义上看略有「领域被挂在存储层」的反向依赖感。  
  - 拆分 `ontology.py` 的初步方向：  
    - 目标：  
      - 保持 `ontology.py` 作为「对外脚本入口」，聚焦 CLI 参数解析与高层操作调度，代码量和逻辑复杂度控制在易读范围；  
      - 将领域逻辑与通用工具从 `ontology.py` 中抽离，形成相对稳定、可复用的核心模块。  
    - 规划中的拆分边界（仅设计，不代表已实施）：  
      - `ontology.py`：  
        - 保留命令行解析、子命令注册与帮助信息；  
        - 负责将 CLI 参数解析为结构化调用（如 `create_entity(...)`、`validate_graph(...)`）并输出结果；  
        - 尽量不直接操作 op 结构或 YAML，只作为「入口壳」。  
      - 新增一个核心脚本模块（例如 `ontology_core.py`，命名待定）：  
        - 集中承载领域操作与适配逻辑，包括：ID 生成、`entity_to_*` / `relation_to_*`、`load_graph`、`create/update/delete` 实体、`create/delete` 关系、`get_related` 以及基于 YAML 的 `validate_graph`；  
        - 统一管理 `BACKEND` 选择与 `get_storage` / `STORAGE` 的初始化，使 CLI 与非 CLI 调用复用同一套核心逻辑；  
        - 对外暴露一组干净的函数接口，供 `ontology.py` 和未来的 helper / 其他脚本调用。  
    - 约束条件与后续准则：  
      - 不额外引入新的目录层级，在现有结构下只增加必要的脚本文件；  
      - 拆分过程中优先做「减法」：先把 `ontology.py` 中明显属于领域 / 工具层的代码迁移到核心模块，而不是继续叠加新功能；  
      - 拆分完成后，`ontology.py` 应保持接近原始项目的脚本风格（专注于 CLI），核心业务与存储细节在其他模块中演进。

- ontology.py 入口脚本进一步瘦身与命令分发表重构
  - 将路径安全校验逻辑 `resolve_safe_path` 下沉至核心模块：  
    - 在 `ontology_core.py` 中统一提供 `resolve_safe_path`，负责限定所有文件路径必须落在工作空间根目录之下，并支持按需校验文件存在性；  
    - `ontology.py` 不再定义该函数，仅在解析完 CLI 参数后调用核心模块的通用工具完成路径解析与校验。  
  - 重构 `main()` 中的命令分发方式：  
    - 保留原有 `argparse` 子命令与参数定义（`create/get/query/list/update/delete/relate/unrelate/related/validate/schema-append` 不发生变更）；  
    - 将原先的大量 `if/elif` 分支拆分为一组独立的 handler 函数（例如 `handle_create` / `handle_get` 等），每个 handler 只关注一种子命令的输入解析与输出逻辑；  
    - 通过 `handlers = { "create": handle_create, ... }` 的字典，采用「command → handler」映射进行统一分发，`main()` 只负责：  
      - 解析参数与路径；  
      - 从映射中查找对应 handler 并执行。  
  - 调整后的职责边界：  
    - `ontology.py`：  
      - 只负责 CLI 层能力（参数定义、路径安全入口、命令 → handler 分发、结果打印），不再持有任何领域 / 存储 / 校验细节；  
    - `ontology_core.py`：  
      - 集中承载所有业务逻辑（领域模型与 op 适配、存储选取与操作、图加载与校验、schema 读写与合并以及通用路径安全工具），成为可复用的核心模块。  
  - 本轮优化的目标是进一步压缩入口脚本的复杂度，使其回归「薄入口 + 粗颗粒调用」的形态，同时不改变任何现有 CLI 行为与输出格式。
